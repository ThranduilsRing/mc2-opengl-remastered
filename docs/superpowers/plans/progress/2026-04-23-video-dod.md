## Delay-load finding (DoD #8)

Investigated via `pefile` inspection of `mc2.exe`:

```
=== Normal imports (only FFmpeg) ===
  avformat-61.dll:  6 symbols
  avcodec-61.dll:  11 symbols
  avutil-59.dll:   11 symbols
  swscale-8.dll:    3 symbols
  swresample-5.dll: 5 symbols
=== Delay imports ===
  (none)
```

The `/DELAYLOAD:*.dll` linker options recorded in `CMakeLists.txt` produce MSVC linker warning **LNK4199** ("no imports found from avcodec-61.dll") at link time. The flags are parsed but dropped — likely because CMake emits the `/DELAYLOAD` options AFTER the FFmpeg import libraries on the command line, at which point the linker has already marked those imports as regular (non-delay).

**Practical impact:** FFmpeg DLLs are hard dependencies. If any is missing from the deploy directory, `mc2.exe` fails to start (exit 127). The `ffmpegProbeAvailability()` function still runs and its failure hook is still installed, but neither is reachable because the process is terminated by the Windows loader before `main()` runs.

**Options for later follow-up:**
1. Reorder CMake so `/DELAYLOAD` + `delayimp.lib` precede the FFmpeg libs on the command line (e.g., via `set_target_properties(mc2 PROPERTIES LINK_FLAGS "/DELAYLOAD:...")` instead of `target_link_options`).
2. Switch to explicit runtime loading via `LoadLibraryA` + `GetProcAddress` (rewrite `mc2video.cpp` to resolve FFmpeg entry points on first use).
3. Accept hard dependency and document the 5 DLLs as required in release notes.

For now, ship option 3. Option 1 is cleaner and worth attempting when a delay-load-required feature arises. Option 2 is the most robust but requires ~100 lines of GetProcAddress boilerplate.

## Concerns noted during implementation

- `/DELAYLOAD` inactive — see above.
- Per-audio-packet `av_malloc`/`av_free` in `decodeNextVideoFrame`. Could be eliminated by pre-allocating a scratch buffer sized to max expected frame samples. Not critical.
- `reinterpret_cast<SoundSystem*>(soundSystem)` pattern in mc2video.cpp (introduced in Task 12) bypasses type system — sound because `GameSoundSystem : public SoundSystem` with no virtual base, but a small API refactor could expose the base pointer cleanly.
- `pefile`-based imports check could live as a release-gate script in `release_assets/` to detect accidental `/DELAYLOAD` breakage if we ever make it work.
